## Runtime & tooling

This project uses **Bun** as both package manager and runtime (see `bun.lock`, `@types/bun`). Use Bun, not npm/yarn/pnpm. The `package.json` scripts invoke `next` directly, so run them through Bun:

```bash
bun install            # install dependencies
bun run dev            # dev server at http://localhost:3000
bun run build          # production build
bun run start          # serve production build
bun run lint           # ESLint (eslint-config-next, core-web-vitals + typescript)
```

There is no test runner configured yet. If adding tests, prefer Bun's built-in test runner (`bun test`).

## Architecture (per SPEC.MD)

Single Next.js (App Router) application serving both frontend and backend. There is no separate API server.

- **Database is Bun's built-in SQLite** (`bun:sqlite`) accessed with **raw SQL** — there is no ORM. The intended data-access layer lives in `lib/db.ts` (singleton connection + `query`/`get`/`run` helpers) and `lib/notes.ts` (note repository functions). DB initialization/migrations are run via a script (e.g. `scripts/init-db.ts`), not generated by a tool.
- **Auth is better-auth.** It manages its own `user`/`session`/`account`/`verification` tables.
- **APIs are Next.js Route Handlers** under `app/api/notes/...` returning JSON.
- **Path alias:** `@/*` maps to the project root (`./`), so import as `@/lib/notes`.

### Two naming conventions coexist in the database — this is intentional

- **better-auth tables use `camelCase`** column names (`emailVerified`, `userId`, `expiresAt`) so better-auth can read/write them with no custom field mapping. Do not rename these.
- **The application `notes` table uses `snake_case`** (`user_id`, `content_json`, `is_public`, `public_slug`, `created_at`, `updated_at`).
- All primary keys (including `notes.id`) are `TEXT` because better-auth generates string IDs.

### Security invariants (non-negotiable)

- Every authenticated note query must filter by `user_id` — the repository functions take `userId` as their first argument precisely to enforce per-user scoping in SQL. Never query notes without it.
- Routes under `/dashboard` and `/notes/[id]` check auth on the server; `/api/notes/*` handlers return `401` when unauthenticated (except public read).
- Note content is stored as **TipTap JSON** (`content_json`), never raw HTML. Render only through TipTap — never `dangerouslySetInnerHTML` with unsanitized data.
- Public sharing works via a random `public_slug` at `/p/[slug]`; disabling sharing sets `is_public = 0` and clears the slug so the URL 404s.

### Rich text

TipTap v3 (`@tiptap/react`, `@tiptap/starter-kit`, `@tiptap/pm`) powers the editor. Content round-trips as TipTap JSON: `JSON.stringify` on save, `JSON.parse` into the editor's `content` on load. Read-only public views use the editor with `editable: false`.
